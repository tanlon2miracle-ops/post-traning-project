# 🚀 风控大模型后训练项目：任务认领与提交 SOP

大家好，为了保证 Phase 1 阶段的高效推进，避免代码冲突和数据混乱，本项目采用**"基于角色的工作区隔离"**与**"PR (Pull Request) 验收制"**。

请大家在开始干活前，务必熟读以下标准工作流：

---

## 🛠️ Step 1：寻找与认领任务

所有待办任务都会以 Markdown 文档的形式发布在 `tasks/` 目录下。

1. **拉取最新代码**：确保你在 `main` 分支，并执行 `git pull`。
2. **寻找任务**：去你所属角色的任务目录下（如 `tasks/01-ops-tasks/`）寻找状态为 `📝 待认领 (Open)` 的任务文档。
3. **创建独立分支**：**千万不要在 main 分支上直接修改！** 根据任务切出一个属于你的工作分支：
   * 命名规范：`{你的角色}/feat-{任务简写}`
   * 例如开发组认领清洗任务：`git checkout -b dev/feat-data-cleaning`
4. **状态打卡**：打开该任务的 `.md` 文件，将顶部的状态修改为 `🚧 进行中 (WIP)`，并写上你的名字（如 `Assignee: @张三`）。然后提交一个 Commit，宣告该任务被你承包。

---

## 💻 Step 2：在专属工作区开发

我们为不同角色的同学划分了物理隔离的代码/数据阵地，**请严格在自己的工作区内干活，不要跨界修改别人的目录**：

* 🟢 **运营/策略同学** 👉 `01-ops-workspace/` （放规则 JSON、少量脱敏样例数据）
* 🔵 **开发/工程同学** 👉 `02-dev-workspace/` （放清洗脚本、部署脚本、环境配置）
* 🟣 **算法大模型同学** 👉 `03-algo-workspace/` （放合成 Prompt、评测脚本、训练 Config）

> ⚠️ **红线警告**：
> 真实的海量人审数据、合成的全量数据，以及下载的模型权重（Checkpoints），**绝对不允许** `git add` 提交到仓库中！请严格遵守 `.gitignore` 的规则。

---

## 📤 Step 3：提交任务与发起 PR (Pull Request)

当你的代码写完或数据准备好，并在本地测试无误后：

1. **推送到远端**：将你的分支 Push 到 Git 仓库。
   ```bash
   git add .
   git commit -m "feat: 完成脱敏数据抽取与规则结构化"
   git push origin {你的分支名}
   ```
2. **发起 PR**：在 Git 平台上，发起一个从你的分支合并到 `main` 分支的 Pull Request (PR)。
3. **填写 PR 描述**：
   * 标题写明完成了什么。
   * 在描述正文中，**必须关联你认领的任务文件**（例如写上：`Closes tasks/01-ops-tasks/TASK-001.md`）。
   * 指定项目负责人（也就是我）作为 Reviewer。

---

## 🔍 Step 4：Review 与合并 (验收)

1. 我会在看到 PR 后进行 Code Review 或数据格式检查。
2. 如果有问题，我会在 PR 里留言，你需要继续在本地原分支修改、Commit 并 Push（PR 会自动更新）。
3. 如果检查通过，我会点击 **Merge** 将你的心血合并入主干。
4. 🎉 任务圆满完成，你可以去 `tasks/` 目录寻找下一个挑战了！

---

## 📋 速查表

| 步骤 | 动作 | 命令 / 位置 |
|------|------|-------------|
| 拉最新代码 | 切到 main 并拉取 | `git checkout main && git pull` |
| 认领任务 | 找到任务文档 | `tasks/{01-ops\|02-dev\|03-algo}-tasks/` |
| 切分支 | 创建工作分支 | `git checkout -b {角色}/feat-{任务}` |
| 打卡 | 修改任务状态 | 任务 .md 顶部改为 `🚧 进行中` |
| 开发 | 在专属工作区干活 | `{01-ops\|02-dev\|03-algo}-workspace/` |
| 提交 | Push 并发 PR | `git push origin {分支}` → 创建 PR |
| 验收 | 等 Review | Reviewer 通过后 Merge 到 main |

## ❓ FAQ

**Q: 我需要修改别人工作区的文件怎么办？**
A: 在 PR 描述中说明原因，并 @相关同学确认。跨区修改需要额外 Review。

**Q: 我不小心把大文件提交了怎么办？**
A: 立刻联系项目负责人。可以用 `git reset` 撤销，但千万不要 force push 到 main。

**Q: 任务做到一半发现方案需要调整？**
A: 在任务文档中记录问题，同时在群里/PR 中讨论。不要闷头改方向。

**Q: docs/ 目录谁来维护？**
A: 所有人可读，但修改需要走 PR。架构性变更由项目负责人统一更新。
