# 技术决策记录

> 所有已讨论并确定的技术决策

## 基座模型

**决策**: Qwen2.5-VL 系列，从 7B 开始验证

| 候选 | 结论 | 原因 |
|------|------|------|
| Qwen3 纯文本 | ❌ 放弃 | 需求是图文多模态，纯文本不够 |
| Qwen2.5-VL-7B | ✅ 一期 | 中文强、社区活跃、VL 原生支持、H20 单卡全参 |
| Qwen2.5-VL-32B | 备选 | 效果更好，H20 多卡 LoRA |
| Qwen2.5-VL-72B | 备选 | 效果天花板 |
| Qwen3-VL | 待观察 | 若发布则评估切换 |

**验证路径**: 7B → 32B → 72B，逐步放大

## 训练框架

**决策**: 复用开源 + 自研数据管道，不自建框架

**理由**:
- 核心竞争力在数据和策略，不在训练框架
- 开源框架成熟稳定，社区持续迭代
- 自研框架工作量大（3-6个月），ROI 低
- 精力集中在有壁垒的地方：数据质量、标签体系、评测

## 技术栈

| 环节 | 工具 | 选型理由 |
|------|------|---------|
| 数据合成（文本） | **Distilabel** (Argilla) | 专做 LLM 数据合成，支持自定义 API endpoint，内置 SFT/DPO pipeline |
| 数据清洗 | **Data-Juicer** (阿里) | 中文友好，80+ 内置算子，支持多模态，配置驱动 |
| 标注质检 | **Argilla** (可选) | 开源标注平台，和 Distilabel 一套 |
| SFT/DPO 训练 | **LLaMA-Factory** | Qwen 官方推荐，支持 VL 多模态，全参/LoRA/QLoRA，内置 SFT+DPO+PPO |
| RL 训练 | **veRL** (字节) | 当前主流 RL 框架，分布式友好（多模态支持待确认） |
| 通用评测 | **lm-evaluation-harness** | 社区标准 |
| 风控评测 | **自研** | 业务特殊性 |
| 推理部署 | **vLLM** | 开箱即用，支持 VL 模型 |

### 放弃的选项

| 工具 | 放弃原因 |
|------|---------|
| NeMo-Curator | 更偏超大规模预训练数据，当前量级用 Data-Juicer 够了 |
| NeMo-Skills | 偏数学/代码，风控场景不直接适用 |
| Self-Instruct | 太老，功能被 Distilabel 覆盖 |
| DataTrove (HF) | 偏预训练数据，后训练不合适 |
| Magpie | 简单但可控性差 |

## 数据策略

### 数据合成

- **内部 API**（GLM / Qwen3），避免数据外泄
- 大模型辅助生成 + 少量人工标注质检
- pass 和 reject 数据都需要，保持平衡
- **图片直接复用**现有人审数据，不需合成图片
- 文本部分为主要合成对象

### 通用能力保留

- SFT 数据混入 **30-40% 通用数据**
- 推荐比例：风控专用 60% + 通用 40%
- 控制 learning rate + epoch 防过拟合

### 质量控制

- **交叉验证**: GLM 合成 → Qwen3 验证（反之亦然）
- **标签一致率**: < 85% 的丢弃或送人审
- **多样性检查**: 语义嵌入聚类
- **难度分布**: easy 40% / medium 40% / hard 20%

## 标签体系

**决策**: 一期 5 个一级标签（MVP）

原始 10+ 细标签区分度不足，先合并验证，后期按需拆分。

详见 [label-taxonomy.md](label-taxonomy.md)

## 训练路线

**决策**: SFT → DPO（可选）→ RL（可选）

- 先跑 SFT，评估效果
- 如果分类任务简单，SFT 可能就够了
- DPO/RL 视 Phase 1 效果决定是否投入
